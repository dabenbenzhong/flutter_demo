# 编辑事项和待办项规格

标签：ready-for-agent

## Problem Statement

当前本地日历已经支持创建、查看、删除和持久化事项，也支持创建、完成、删除和持久化待办项，但用户一旦输错标题、时间、备注或颜色，只能删除原数据后重新创建。这会打断原有数据状态，也使日常修正内容变得繁琐。应用需要补齐已有事项和待办项的编辑能力，让本地数据形成完整、直接且可预期的维护闭环。

## Solution

在事项和待办项现有展示位置提供明确的编辑入口。编辑操作使用与新增操作一致的底部表单和校验规则，并预填原数据。用户保存后，原数据被修改后的内容替换，各功能页面立即显示一致结果并写入本地存储；用户取消编辑时不产生任何变化。

事项本轮允许修改标题、开始时间、结束时间、备注和颜色，不允许修改所属日期。待办项允许修改标题和备注，并保留原有完成状态和创建时间。编辑功能不引入提醒、搜索、历史记录、撤销或批量操作。

## User Stories

1. 作为本地日历用户，我想编辑已有事项，以便修正创建时输入错误的内容。
2. 作为本地日历用户，我想从日历页当天事项列表进入编辑，以便在查看当天安排时直接修改事项。
3. 作为本地日历用户，我想从日程页进入事项编辑，以便在跨日期浏览安排时直接修改事项。
4. 作为本地日历用户，我想看到明确的“编辑事项”入口，以便不需要猜测长按或点击整行等隐藏手势。
5. 作为本地日历用户，我想编辑表单预填事项原有标题，以便只修改需要调整的部分。
6. 作为本地日历用户，我想编辑表单预填事项原有开始时间和结束时间，以便保留正确的时间信息。
7. 作为本地日历用户，我想编辑表单预填事项原有备注，以便继续维护已有上下文。
8. 作为本地日历用户，我想编辑表单预选事项原有颜色，以便不修改颜色时保持原样。
9. 作为本地日历用户，我想修改事项标题，以便修正或更新事项名称。
10. 作为本地日历用户，我想修改事项开始时间和结束时间，以便安排发生变化时更新当天时间。
11. 作为本地日历用户，我想修改事项备注，以便补充、更新或清除轻量上下文。
12. 作为本地日历用户，我想修改事项颜色，以便调整事项的视觉区分。
13. 作为本地日历用户，我想编辑事项时继续看到其所属日期，以便确认正在修改哪一天的事项。
14. 作为本地日历用户，我不想在本轮编辑事项时改变所属日期，以便编辑流程保持简单且不意外移动事项。
15. 作为本地日历用户，我想事项标题仍然必填，以便编辑后不会出现无法识别的事项。
16. 作为本地日历用户，我想事项开始时间和结束时间仍然必填，以便编辑后事项时间保持完整。
17. 作为本地日历用户，我想编辑表单继续校验 HH:mm 时间格式，以便不会保存无效时间。
18. 作为本地日历用户，我想结束时间必须晚于开始时间，以便编辑不会破坏事项时间规则。
19. 作为本地日历用户，我想修正无效字段后错误提示及时消失，以便清楚知道内容已经有效。
20. 作为本地日历用户，我想保存编辑后立即看到更新后的事项，以便确认修改已经生效。
21. 作为本地日历用户，我想事项时间变化后当天事项重新按开始时间排序，以便列表仍按时间顺序阅读。
22. 作为本地日历用户，我想从日历页编辑事项后日程页同步显示更新，以便不同功能页面看到同一份本地数据。
23. 作为本地日历用户，我想从日程页编辑事项后日历页同步显示更新，以便不同入口不会产生不一致副本。
24. 作为本地日历用户，我想编辑事项后事项数量保持不变，以便编辑不会被误认为新增。
25. 作为本地日历用户，我想取消事项编辑后原事项完全不变，以便可以安全退出表单。
26. 作为本地日历用户，我想关闭并重新打开应用后仍看到编辑后的事项，以便修改结果被持久保存。
27. 作为本地日历用户，我想原有事项删除入口继续独立存在，以便编辑和删除是两个清晰操作。
28. 作为本地日历用户，我不想保存普通编辑时再确认一次，以便高频修改保持直接。
29. 作为本地日历用户，我想编辑已有待办项，以便修正标题或备注而无需删除重建。
30. 作为本地日历用户，我想在待办页看到明确的“编辑待办项”入口，以便操作容易发现。
31. 作为本地日历用户，我想编辑表单预填待办项原有标题和备注，以便只修改需要调整的内容。
32. 作为本地日历用户，我想修改待办项标题，以便更新待处理内容。
33. 作为本地日历用户，我想修改、补充或清空待办项备注，以便维护轻量上下文。
34. 作为本地日历用户，我想待办项标题在编辑时仍然必填，以便待办列表不出现空内容。
35. 作为本地日历用户，我想编辑完成的待办项后仍保持完成状态，以便文字修正不会改变进度。
36. 作为本地日历用户，我想编辑未完成的待办项后仍保持未完成状态，以便文字修正不会意外完成待办。
37. 作为本地日历用户，我想编辑待办项后保留原创建时间，以便同组排序语义不因编辑而改变。
38. 作为本地日历用户，我想保存编辑后立即看到更新后的待办项，以便确认修改已经生效。
39. 作为本地日历用户，我想编辑待办项后待办数量保持不变，以便编辑不会生成副本。
40. 作为本地日历用户，我想取消待办项编辑后原待办项完全不变，以便可以安全退出表单。
41. 作为本地日历用户，我想关闭并重新打开应用后仍看到编辑后的待办项，以便修改结果被持久保存。
42. 作为本地日历用户，我想编辑待办项后原有完成切换和删除操作继续可用，以便现有维护能力不受影响。
43. 作为使用辅助技术的用户，我想事项和待办项的编辑按钮有清晰的可访问名称，以便知道按钮会编辑哪条数据。
44. 作为使用辅助技术的用户，我想编辑表单字段继续暴露清晰的业务语义名称，以便理解当前输入内容。
45. 作为窄屏设备用户，我想新增编辑按钮后列表内容仍不溢出或被遮挡，以便小屏幕上仍可完整操作。
46. 作为本地日历用户，我想已有本地数据无需迁移或重新创建就能编辑，以便升级后继续使用原数据。
47. 作为本地日历用户，我想编辑功能保持当前应用的视觉和交互语言，以便不需要学习一套新表单。

## Implementation Decisions

- 本阶段只实现编辑事项和编辑待办项，不同时实现提醒、搜索或其他新业务能力。
- 事项允许修改标题、开始时间、结束时间、备注和颜色。
- 事项所属日期在本阶段不可修改；“移动事项”是后续独立能力。
- 待办项允许修改标题和备注。
- 编辑待办项必须保留原完成状态和原创建时间。
- 日历页当天事项列表和日程页事项列表都提供明确的事项编辑按钮。
- 待办页的每个待办项提供明确的编辑按钮。
- 不以长按、双击或点击整行作为唯一编辑入口，避免隐藏交互和可访问性问题。
- 编辑复用现有底部表单视觉和字段组件；表单根据新增或编辑模式显示对应标题和提交语义。
- 编辑表单打开时预填原数据；事项颜色控件预选原颜色。
- 编辑事项时显示所属日期，但不提供修改日期的控件。
- 新增和编辑继续共享同一套字段校验规则，避免同一数据在不同入口产生不同合法性标准。
- 保存编辑会在当前应用状态中替换原对象，而不是追加新对象。
- 保存后继续通过现有应用级本地数据边界整体持久化事项和待办项。
- 事项时间发生变化后，所有按事项时间展示的列表继续依据修改后的开始时间排序。
- 编辑保存后，日历页、日程页、待办页和我的页统计自然反映同一份本地数据；事项和待办项数量不变化。
- 取消或关闭编辑表单不会修改内存状态，也不会触发持久化写入。
- 删除继续使用现有独立入口和二次确认；普通编辑保存不增加二次确认。
- 本阶段不增加稳定 ID，也不改变现有 JSON 数据结构。当前单设备应用通过列表中的原对象定位并替换数据，足以满足编辑需求。
- 现有旧数据保持可读取和可编辑，不需要用户执行数据迁移。
- 若未来引入跨对象引用、同步、编辑历史或并发更新，再单独设计稳定身份和数据迁移策略。
- 编辑入口、输入字段和状态控件必须保留或增加清晰的语义标签。
- 新增编辑按钮后必须保持窄屏布局可用，不得让标题、时间、备注或操作入口溢出、重叠或被裁切。
- 本阶段不新增 ADR；当前决定范围可控、易于后续扩展，不属于难以逆转的架构选择。

## Testing Decisions

- 测试关注用户可观察行为，不断言私有 widget、控制器或列表替换实现。
- 主要测试缝继续使用最高层应用 widget，因为现有测试已通过这一入口覆盖底部导航、事项和待办操作、跨页面同步、表单校验及持久化可见行为。
- 存储层继续使用现有应用级本地数据存储缝，验证编辑后的事项和待办项可以完成 JSON round-trip，并且旧格式数据仍能加载。
- 事项编辑测试覆盖：从日历页进入编辑、字段预填、保存修改、取消不修改、必填校验、时间格式校验以及结束时间规则。
- 事项编辑测试覆盖：修改开始时间后当天列表重新排序。
- 事项编辑测试覆盖：从日历页编辑后日程页同步，以及从日程页编辑后日历页同步。
- 事项编辑测试覆盖：编辑不会改变所属日期或事项数量。
- 事项编辑测试覆盖：重新构建或重新加载应用后编辑结果仍存在。
- 待办项编辑测试覆盖：字段预填、保存修改、取消不修改和标题必填。
- 待办项编辑测试覆盖：编辑前后的完成状态和创建时间保持不变。
- 待办项编辑测试覆盖：编辑不会改变待办项数量或同组排序语义。
- 待办项编辑测试覆盖：重新构建或重新加载应用后编辑结果仍存在。
- 可访问性测试覆盖事项编辑按钮、待办项编辑按钮和编辑表单字段的语义名称。
- 移动端尺寸测试覆盖新增编辑入口后的事项行和待办行，确保窄屏下无溢出、重叠或遮挡。
- 现有新增、删除、完成切换、清空数据、月份切换、四页导航和本地持久化测试必须继续通过。
- 完成后运行全部 widget 测试、存储测试和主题测试，并运行静态分析。
- 修改包含中文文本的代码和文档后，按项目规则检查 UTF-8 和中文乱码。

## Out of Scope

- 修改事项所属日期或把事项移动到其他日期。
- 复制事项或复制待办项。
- 批量编辑事项或待办项。
- 撤销、重做或编辑历史。
- 自动保存草稿。
- 编辑冲突和并发更新处理。
- 稳定 ID、数据库迁移或存储格式升级。
- 提醒和通知。
- 重复事项。
- 搜索、筛选、分类或标签。
- 待办项截止日期、优先级、提醒或重复。
- 账号、云同步、备份或多设备一致性。
- 系统日历接入。
- 全天事项或跨天事项。
- 新增深色模式、国际化、品牌、图标或启动页。

## Further Notes

本规格建立在已经完成的本地日历 MVP、四功能页面、应用级 UI 设计系统和 Android QA 修复之上。实现应优先复用现有表单、共享 UI 组件、应用级状态和本地数据边界，避免为编辑能力建立平行路径。

领域词汇已经在 `CONTEXT.md` 中补充“编辑事项”和“编辑待办项”。事项编辑明确不等同于移动事项，待办项编辑明确不等同于切换完成状态。
