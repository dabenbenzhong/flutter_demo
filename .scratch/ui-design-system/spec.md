# 应用级 UI 设计系统规格

标签：ready-for-agent

## Problem Statement

当前本地日历已经具备日历页、日程页、待办页和我的页的核心功能，但 UI 没有统一的设计基础。颜色、字体样式、间距、圆角、阴影和组件形态散落在各个功能页面和 widget 中，导致页面之间像是逐步拼接出来的，而不是同一个精致产品的一部分。用户希望项目继续变完整，但下一阶段最重要的问题不是新增业务功能，而是先把现有四个功能页面做成统一、稳定、漂亮的本地日历体验。

## Solution

建立全应用级 UI 基础层，用统一的设计 token 和共享组件承载“温暖、克制、精致的本地日历工具”视觉方向。应用通过 `ThemeData` 和 `ThemeExtension` 管理颜色、文字层级、间距尺度、圆角、阴影和组件风格，四个功能页面逐步迁移为消费统一 theme token 和共享组件，不再各自散写样式细节。第一轮做中等深度 UI 重构：保持现有业务功能、页面结构和主交互路径不变，集中统一视觉基础并精修日历页、日程页、待办页和我的页的一致性。

## User Stories

1. 作为本地日历用户，我想四个功能页面看起来属于同一个应用，以便使用时不会感到风格割裂。
2. 作为本地日历用户，我想日历页、日程页、待办页和我的页使用一致的颜色语言，以便快速理解哪些元素是主要操作、次要信息和状态提示。
3. 作为本地日历用户，我想页面标题、正文、说明文字和数字统计有稳定的文字层级，以便内容更容易扫描。
4. 作为本地日历用户，我想各页面的留白和组件间距保持一致，以便界面更精致、更有秩序。
5. 作为本地日历用户，我想按钮、卡片、空状态和底部弹窗的视觉形态一致，以便每个操作都符合预期。
6. 作为本地日历用户，我想现有创建事项、查看事项、管理待办和清空本地数据的路径保持不变，以便 UI 精修后不需要重新学习应用。
7. 作为本地日历用户，我想日历页仍然突出月份网格和当天事项，以便核心日历体验不被装饰性内容干扰。
8. 作为本地日历用户，我想日程页的跨日期事项列表更清晰，以便按时间线查看安排。
9. 作为本地日历用户，我想待办页的未完成和已完成状态更容易区分，以便快速判断待处理内容。
10. 作为本地日历用户，我想我的页的数据统计和清空入口更有层次，以便理解这是本地数据管理区域。
11. 作为本地日历用户，我想空状态在日程页和待办页中表现一致，以便知道当前没有数据以及下一步可以做什么。
12. 作为本地日历用户，我想二次确认弹窗使用一致的标题、正文和按钮风格，以便风险操作更明确。
13. 作为本地日历用户，我想新增事项和新增待办的底部表单使用一致的布局和字段样式，以便填写体验稳定。
14. 作为本地日历用户，我想底部导航的选中态和未选中态清楚统一，以便知道当前处于哪个功能页面。
15. 作为本地日历用户，我想移动端窄屏下文字不会拥挤或溢出，以便小屏使用也保持完整。
16. 作为本地日历用户，我想页面不要堆叠过多装饰性卡片，以便主要内容更直接。
17. 作为本地日历用户，我想温暖视觉风格保留下来，但不要只有单一棕橙色调，以便界面更耐看。
18. 作为本地日历用户，我想事项颜色和状态颜色有稳定语义，以便颜色不只是随机装饰。
19. 作为本地日历用户，我想操作反馈和校验提示保持统一，以便知道哪里需要修正。
20. 作为本地日历用户，我想 UI 精修后已有数据和本地持久化行为不受影响，以便放心继续使用。
21. 作为开发者，我想新增 UI 时能从统一 theme 读取颜色、文字、间距、圆角和阴影，以便不再复制散落样式。
22. 作为开发者，我想共享组件覆盖常见页面块，以便日历页、日程页、待办页和我的页能复用同一套结构。
23. 作为开发者，我想保留高层 widget 测试作为主要回归保护，以便 UI 重构不破坏用户可见行为。
24. 作为开发者，我想设计 token 能通过 Flutter theme 扩展承载，以便未来支持深色模式或主题调整时有清晰入口。
25. 作为开发者，我想第一轮重构范围明确，以便先解决 UI 不统一问题，而不是同时扩展新业务功能。

## Implementation Decisions

- 下一阶段聚焦 UI 统一和视觉精修，不扩展提醒、重复事项、搜索、分类、账号、云同步等新业务能力。
- 视觉方向收敛为温暖、克制、精致的本地日历工具。
- 采用全应用级 UI 基础层，而不是只在日历功能内定义局部样式。
- 技术入口采用 `ThemeData` 加 `ThemeExtension`，用于统一管理颜色、字体、间距、圆角、阴影和组件风格。
- 第一轮采用中等深度重构：不推翻现有四个功能页面的信息架构和主交互路径，但要把视觉基础统一到设计系统。
- 颜色应从原始十六进制值迁移到语义 token，例如背景、表面、主要文字、次要文字、边框、主操作、危险操作、事项标记和状态色。
- 文字样式应形成稳定层级，例如页面标题、区块标题、卡片标题、正文、辅助说明、数字统计、按钮和底部导航文本。
- 间距应形成有限尺度，页面外边距、组件内边距、区块间距和列表项间距不能继续各自随意取值。
- 圆角应形成有限尺度，并降低过度圆润的卡片感；普通内容卡片保持克制，只有浮动按钮、图标容器或胶囊控件使用更大的圆角。
- 阴影应形成轻量语义，避免每个组件自定义不一致的模糊、透明度和偏移。
- 页面背景应统一，保留温暖感，但减少散落的装饰性绘制和过重的视觉噪音。
- 共享组件应覆盖页面容器、页面标题、内容卡片、空状态、统计行、确认弹窗、底部表单、主操作按钮和图标按钮等常见 UI。
- 日历页应继续突出当前显示月份、月份网格、选中日期、当天事项和新增事项入口。
- 日程页应使用统一页面标题、空状态、日期分组和事项列表样式，不新增事项表单。
- 待办页应使用统一页面标题、空状态、待办列表项、完成状态和删除入口样式。
- 我的页应使用统一页面标题、说明、统计和危险操作样式，不引入账号、云同步、会员或备份语义。
- 新增事项和新增待办的底部表单应统一标题、字段、校验提示、按钮和底部安全区处理。
- 二次确认弹窗应统一风险操作语言和按钮层级，危险确认操作要和普通主操作有所区分。
- 底部导航应保留四个功能页面入口，并统一选中态、未选中态、触控区域、图标大小和文字样式。
- 设计系统迁移后，feature 页面不应继续新增孤立的 `Color`、`TextStyle`、`EdgeInsets`、阴影或圆角常量；必要的特殊值应先判断是否能成为 token。
- 已有本地数据、事项、待办项、月份切换、删除确认、清空本地数据和持久化行为保持不变。
- 本轮只实现浅色主题；`ThemeExtension` 的结构要为未来主题扩展保留清晰入口，但不交付深色模式。
- 本轮不做品牌命名、应用图标、启动页或营销型首页。

## Testing Decisions

- 测试应关注用户可观察行为和稳定视觉约束，不断言私有 widget 的内部实现细节。
- 主要测试切入点继续使用最高层 `CalendarApp` widget，因为它能覆盖四个功能页面、底部导航、底部表单、确认弹窗和跨页面数据联动。
- 现有本地日历功能测试应继续通过，证明 UI 重构没有破坏事项、待办项、本地数据、月份切换和清空本地数据行为。
- 新增或调整 widget 测试，覆盖四个功能页面在重构后仍可进入，且关键文案、空状态、列表内容和操作入口仍存在。
- 新增或调整移动端尺寸测试，覆盖窄屏下页面标题、按钮文字、底部导航和卡片内容不溢出、不重叠。
- 对设计 token 做轻量测试或断言，确认应用主题能提供所需的颜色、文字、间距、圆角和阴影扩展。
- 对共享组件做聚焦 widget 测试，优先覆盖空状态、统计块、确认弹窗和底部表单这类跨页面复用组件。
- 不使用测试去锁死每一个像素值；只有会影响可读性、触控区域、溢出或跨页面一致性的约束才进入测试。
- 视觉精修后运行 `flutter test`，确认全部行为测试通过。
- 视觉精修后运行 `flutter analyze`，确认没有静态分析问题。
- 修改包含中文文本的代码和文档后，按项目规则检查 UTF-8/中文乱码。

## Out of Scope

- 新增提醒和通知。
- 新增重复事项。
- 新增搜索、筛选、分类或标签。
- 新增账号、登录、云同步、备份、会员或多设备一致性。
- 读取或写入系统日历。
- 编辑已有事项或待办项。
- 待办项截止日期、优先级、分类、提醒或重复。
- 品牌重命名、应用图标、启动页和营销型首页。
- 深色模式。
- 国际化和多语言切换。
- 自定义字体引入。
- 动画系统重构。
- 数据存储结构迁移。
- 大幅调整四个功能页面的信息架构或主交互路径。

## Further Notes

本规格建立在已完成的四功能页面应用之上，并遵守已记录的“采用应用级设计系统统一 UI”架构决策。实现时应先建立设计系统的最小可用骨架，再按页面迁移，避免只做局部视觉修补。完成后再拆分实现 tickets，优先处理 theme token 和共享组件，然后迁移日历页、日程页、待办页、我的页，最后做整体视觉和乱码检查。
