# 本地日历 MVP 规格

标签：ready-for-agent

## 问题陈述

当前日历应用还是静态演示：日期选中态、当天事项列表和新增按钮都只是视觉呈现，用户无法选择日期、创建事项、删除事项，也无法在退出应用后保留自己的数据。用户需要一个真正可用的本地日历 MVP，在当前设备内记录每天的事项，并能围绕选中日期查看当天安排。

## 解决方案

将现有静态日历演示推进为本地日历 MVP。用户可以在当前月份网格中点选日期，查看选中日期的当天事项；点击右下角加号打开底部表单，创建包含标题、日期、开始时间、结束时间、备注和颜色的事项；事项保存到当前设备，退出应用后仍保留；用户可以删除事项，删除前需要二次确认。

保留现有温暖卡片风格、月视图、底部导航和右下角加号。底部导航在 MVP 中仍作为视觉壳存在，不实现多页面。

## 用户故事

1. 作为日历用户，我想打开应用后仍看到现有日历仪表盘风格，以便 MVP 延续当前产品观感。

3. 作为日历用户，我想点击当前月份网格中的某一天，以便把它设为选中日期。
4. 作为日历用户，我想点击其他日期后选中态随之移动，以便清楚知道正在查看哪一天。
5. 作为日历用户，我想在事项区域看到选中日期，以便理解列表上下文。
6. 作为日历用户，我想事项列表只显示选中日期的事项，以便不被其他日期干扰。
7. 作为日历用户，我想在没有事项时看到“当天没有事项”，以便知道选中日期没有安排。
8. 作为日历用户，我不想在没有本地事项时看到假的演示事项，以便信任页面显示的数据。
9. 作为日历用户，我想使用现有加号按钮创建事项，以便主要创建入口清晰可见。
10. 作为日历用户，我想点击加号后打开底部表单，以便不离开日历页就能新增事项。
11. 作为日历用户，我想创建表单默认使用选中日期，以便新事项自动归属到正在查看的那一天。
12. 作为日历用户，我想输入事项标题，以便在事项列表中识别它。
13. 作为日历用户，我想输入开始时间，以便事项有明确的开始。
14. 作为日历用户，我想输入结束时间，以便事项有明确的结束。
15. 作为日历用户，我想输入可选备注，以便保存轻量上下文。
16. 作为日历用户，我想选择颜色，以便用视觉方式区分事项。
17. 作为日历用户，我想应用拒绝结束时间不晚于开始时间的事项，以便不会保存不合理的事项时间。
18. 作为日历用户，我想在必填字段缺失或无效时看到校验反馈，以便知道保存前要修正什么。
19. 作为日历用户，我想保存成功后立即在选中日期的事项列表中看到新事项，以便确认它已创建。
20. 作为日历用户，我想事项按开始时间升序排列，以便当天安排按时间顺序阅读。
21. 作为日历用户，我想月历网格标记有事项的日期，以便快速扫出哪些天有安排。
22. 作为日历用户，我想关闭并重新打开应用后事项仍然存在，以便本地日历不只在单次会话内有效。
23. 作为日历用户，我想删除事项，以便移除误建或过期的本地数据。
24. 作为日历用户，我想删除前看到确认弹窗，以便误触不会立即移除事项。
25. 作为日历用户，我想取消删除确认后事项保持不变，以便可以安全返回。
26. 作为日历用户，我想确认删除后事项从选中日期列表消失，以便可见列表反映我的选择。
27. 作为日历用户，我想底部导航保持视觉一致，以便 MVP 不像一个完全不同的应用。
28. 作为日历用户，我想 MVP 避免账号、同步和系统日历权限，以便无需额外设置即可本地使用。

## 实现决策

- MVP 是本地日历：数据只保存在当前设备，退出应用后仍保留，不使用账号、云同步或系统日历。
- 领域术语使用“事项”；不要把它混同为任务、待办或外部日历事件。
- 事项包含标题、日期、开始时间、结束时间、备注和颜色。
- 地点、提醒、重复、参与人、附件、优先级、全天事项和跨天事项不进入 MVP。
- 事项发生在单个日期内。结束时间必须晚于开始时间。
- 事项列表始终以选中日期为范围，并按开始时间升序排列。
- 日期选择只限当前月份网格中可见的日期。月份切换、年份选择和跳转今天不进入范围。
- 加号按钮打开底部表单。表单使用选中日期作为事项日期，表单内不提供改日期能力。
- 保存有效表单后创建事项、关闭底部表单，并刷新选中日期的事项列表。
- 删除事项需要确认弹窗。确认后删除；取消后保持不变。
- 选中日期没有事项时显示“当天没有事项”。
- 用户可见的事项列表不再使用演示事项数据，而是反映真实本地状态。
- 本地持久化先使用轻量键值存储保存 JSON 列表，不引入数据库。
- 存储边界要足够窄，后续替换为数据库时不应改变用户可见行为。
- 保留现有视觉系统：温暖卡片风格、月视图、底部导航壳和右下角加号。
- 底部导航在 MVP 中仍是视觉壳，不创建多个功能页面。

## 测试决策

- 测试应关注用户可观察行为，而不是实现细节。通过的测试应证明用户能看到什么、做什么，而不是证明调用了哪个内部存储类。
- 主要测试切入点是日历应用或日历首页 widget，因为这是覆盖日期选择、事项创建、校验、删除、空状态、排序和持久化可见行为的最高有效切入点。
- 现有 widget 测试已经覆盖静态仪表盘、月历网格、选中日期渲染、移动端紧凑布局和关键视觉文案；实现时应把这些静态断言调整为有状态用户行为断言。
- 增加点击日期并验证选中日期和事项内容变化的 widget 测试。
- 增加选中日期无事项时显示“当天没有事项”的 widget 测试。
- 增加点击加号打开底部表单的 widget 测试。
- 增加填写有效事项表单、保存并在选中日期列表看到事项的 widget 测试。
- 增加按开始时间升序排列的 widget 测试。
- 增加结束时间不晚于开始时间时显示校验反馈的 widget 测试。
- 增加删除确认的取消路径和确认路径 widget 测试。
- 增加一个面向持久化的测试，使用可替换的本地存储边界或测试存储实现，验证保存后的事项能被新的应用状态加载。

## 不在范围内

- 编辑已有事项。
- 月份切换、年份选择和跳转今天。
- 账号、登录、云同步、备份或多设备一致性。
- 读取或写入系统日历。
- 提醒和通知。
- 重复事项。
- 地点、参与人、附件、优先级和标签。
- 全天事项和跨天事项。
- 搜索、筛选或分类管理。
- 实现底部导航对应的功能页面。
- 迁移到 SQLite、Drift 或其他数据库。
